# 01 项目定位（Project Positioning）

> ⚠️ **2026-05 重构后**：本文提及的 deepagents/langgraph 框架已被替换为纯 Python
> agent loop（`agent/loop.py`）。功能不变，技术栈细节看 `CLAUDE.md` / [`docs/README.md`](../README.md) 顶部映射表。

## 一、一句话定位

**华为广告平台数据分析 AI Agent** —— 一个让营销 / 运维 / 数据分析师通过自然语言对话，
完成数据查询、异常诊断、深度分析、报表生成的智能助手框架。

> ⚠️ **当前阶段聚焦"分析场景"，不涉及任何系统写操作**（投放修改、活动暂停、预算调整等）。
> 所有输出是**只读分析结果**——表格、图表、诊断结论、文字报告。
> 框架代码已实现 HitL 写操作守卫机制，预留给未来需要时启用，**当前业务场景不依赖**。

## 二、目标用户

| 角色 | 典型诉求（仅分析场景） |
|---|---|
| 营销专员 | 查活动报表、看 CTR / 转化率 / ROI 异动、生成周报与日报、跨周期对比 |
| 运维 | 监控异常诊断、运行状态查询、问题活动定位、链路瓶颈分析（**只读**——不涉及任何配置/维护/操作） |
| 数据分析师 | **深度广告数据分析**——指标钻取、人群画像、行为分析、跨维度交叉、自定义分析逻辑（上传 SKILL.md） |

不针对：

- ❌ 终端 C 端用户（广告受众）
- ❌ 广告主自助平台（外部客户）
- ❌ **任何系统写操作**：投放策略调整、活动暂停 / 删除、预算修改、出价调整等
- ❌ 实时投放决策系统（自动出价 / 自动调整投放策略）
- ❌ 广告主侧或端侧的配置 / 维护工具

## 三、核心使用场景

### 3.1 自然语言查报表

用户："查询昨天中国区 KA 客户的 CTR 异常数据"
→ Agent 调 `query-metrics` 工具，输出表格 + 折线图 + 异常诊断建议

### 3.2 多轮交互式诊断

用户：「上面那个 CTR 跌得最多的客户，看下它的曝光分布和转化漏斗」
→ Agent 通过 conversation 上下文记住"那个客户"是谁，自动展开下钻

### 3.3 深度分析（数据分析师）

用户：「分析过去 30 天高价值人群（LTV>1000）的曝光分布与转化漏斗，找出转化率低的环节」
→ Agent 调 `query-metrics` + `query-audience-profile` + `funnel-analysis` 多个 skill，输出
跨维度对照表 + 漏斗图 + 异常环节诊断

### 3.4 异常诊断（运维）

用户：「中国区今早 9 点开始 KA 客户的 CTR 突降，给我定位下哪些客户 / 活动出了问题」
→ Agent 调 `query-metrics`（按时段对比）+ `detect-anomaly` + `query-campaign-detail`，
输出问题客户清单 + 异常时间线 + 可能原因猜测——**只输出诊断结论，不执行任何操作**

### 3.5 自定义分析能力

用户上传自己写的 SKILL.md 包（如"按行业分组对比"、"自定义人群标签交叉")
→ 立即可在对话里调用

## 四、与传统 BI / 仪表盘工具的差异点

| 维度 | 传统 BI（Tableau / Quick BI / 自研报表） | 本 Agent |
|---|---|---|
| 入口 | 拖拽配置仪表盘 | 自然语言对话 |
| 灵活性 | 预定义 dashboard | 任意临时问题，按需查 |
| 跨数据源 | 需 ETL 预处理 | LLM 编排多个 skill 调用 |
| 异常诊断 | 看图自己判断 | LLM 自动诊断并解释 |
| 操作能力 | 只读 | 只读（当前阶段；框架预留 HitL 写操作能力） |
| 学习成本 | 中（需学拖拽 + 配置） | 低（会说话就行） |
| 适合的问题 | 高频 / 标准化报表 | 低频 / 临时性 / 探索性 |

**Agent 不替代 BI，是补充**——高频标准报表仍走 dashboard，临时探索 + 跨数据源 + 异常诊断走 Agent。

## 五、产品边界（不做什么）

- ❌ **当前阶段不做任何系统写操作**：Agent 仅产出分析结果（表格、图表、诊断、报告），所有"删除 / 暂停 / 修改预算 / 调整出价"等写操作**不在范围**——交由现有运营平台执行
- ❌ **不做实时投放决策系统**：不自动改投放策略
- ❌ **不做广告主自助平台**：当前只服务**内部**营销 / 运维 / 数据分析师；广告主侧自助产品不在范围
- ❌ **不做端侧或广告主配置维护**：投放参数、素材审核、账户开通等由现有平台负责，Agent 不接入
- ❌ **不做仪表盘**：高频报表请用现有 BI 工具，Agent 服务的是临时性 / 探索性 / 跨维度分析
- ❌ **不做端到端归因模型**：Agent 调用现有归因 skill，不做模型训练
- ❌ **不做多语言**：仅中文 prompt + 中文回复，UI 也是中文（华为内部使用场景）

> 💡 **HitL 框架能力**已实现并测试通过（见 [02-features/04-human-in-the-loop.md](../02-features/04-human-in-the-loop.md)），
> 当未来业务扩展到写操作时（如批量暂停异常活动），只需 SKILL.md 包注册写工具 +
> `config.yaml::agent.interrupt_on` 列出即可启用，**框架代码无需改动**。

## 六、技术定位

本仓库是一个**通用框架**——LLM 编排 + Channel 抽象 + SKILL.md 加载 + HitL 确认 + 持久化等
"通用 Agent 平台"能力。具体业务能力（指标查询 / 报表导出 / 异常检测）通过 SKILL.md 包外挂。

→ 同样的框架代码可以服务："广告 Data Agent"、"运维 Data Agent"、"客服 Data Agent" 等
任何 *数据分析 + 自然语言* 场景。换 SKILL.md 包就行。

详见 [02-features/01-skill-tool-channel-model.md](../02-features/01-skill-tool-channel-model.md)。

## 七、与同类项目的关系

| 项目 | 关系 |
|---|---|
| `langchain` / `langgraph` | 我们站在它的肩膀上（langgraph 提供 agent 状态机 + checkpointer） |
| `deepagents` | langgraph 之上的 agent 装配 SDK，我们直接复用其 SubAgentMiddleware / SummarizationMiddleware 等能力 |
| `Claude Code` | mental model 参考——Skill / Tool / SubAgent 的概念分层向它对齐 |
| `Anthropic Agent SDK` | 类似但更"产品化"——本项目是定制版（含中文 / 华为广告域 prompt） |
| `FastAPI` / `Vue 3` | 业务接入框架（HTTP / SPA） |

不是 fork / 替代品 —— 我们是一个**面向广告数据场景的 agent 平台实现**，借助上述生态搭建。
